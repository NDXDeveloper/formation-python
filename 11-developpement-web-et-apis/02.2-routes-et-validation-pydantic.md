# 11.2.2 Routes et validation avec Pydantic

## Introduction

Dans la section précédente, vous avez créé votre première API FastAPI avec des routes simples. Maintenant, nous allons aller plus loin en découvrant **Pydantic**, l'une des fonctionnalités les plus puissantes de FastAPI. Pydantic permet de valider automatiquement les données que vous recevez et envoyez, rendant votre API robuste et fiable.

## Qu'est-ce que Pydantic ?

**Pydantic** est une bibliothèque Python qui permet de valider des données en utilisant les annotations de type Python. C'est comme avoir un garde du corps pour vos données : il vérifie que tout est conforme avant de laisser entrer quoi que ce soit dans votre application.

### Analogie simple

Imaginez un formulaire d'inscription sur un site web :
- Sans validation : N'importe qui peut écrire n'importe quoi (un âge négatif, un email invalide, etc.)
- Avec Pydantic : Le formulaire vérifie automatiquement que l'email est valide, que l'âge est un nombre positif, etc.

Pydantic fait cela pour vous, automatiquement, en utilisant simplement les annotations de type Python !

## Pourquoi Pydantic est essentiel avec FastAPI

FastAPI utilise Pydantic pour :

1. **Valider automatiquement** les données entrantes
2. **Convertir** les données dans les bons types
3. **Générer automatiquement** la documentation de l'API
4. **Fournir des messages d'erreur clairs** quand les données sont invalides
5. **Assurer la cohérence** des données dans toute votre application

Le meilleur ? Vous n'avez presque rien à faire ! Il suffit de définir vos modèles de données, et FastAPI + Pydantic s'occupent du reste.

## Installation de Pydantic

Bonne nouvelle : Pydantic est automatiquement installé avec FastAPI ! Vous n'avez rien de plus à installer.

Si toutefois vous voulez vérifier :

```bash
pip list | grep pydantic
```

Vous devriez voir `pydantic` dans la liste.

## Les modèles Pydantic : BaseModel

Un **modèle Pydantic** est une classe Python qui hérite de `BaseModel`. C'est dans cette classe que vous définissez la structure de vos données.

### Votre premier modèle

Créons un modèle simple pour représenter un utilisateur :

```python
from fastapi import FastAPI  
from pydantic import BaseModel  

app = FastAPI()

# Définir un modèle Pydantic
class Utilisateur(BaseModel):
    nom: str
    age: int
    email: str
```

C'est tout ! Vous avez créé un modèle qui définit qu'un utilisateur doit avoir :
- Un `nom` (chaîne de caractères)
- Un `age` (nombre entier)
- Un `email` (chaîne de caractères)

### Utiliser le modèle dans une route POST

Maintenant, utilisons ce modèle dans une route pour créer un utilisateur :

```python
@app.post("/utilisateurs")
def creer_utilisateur(utilisateur: Utilisateur):
    return {
        "message": "Utilisateur créé avec succès",
        "utilisateur": utilisateur
    }
```

Que se passe-t-il ici ?

1. Le client envoie des données JSON dans le corps de la requête
2. FastAPI utilise Pydantic pour **valider** ces données
3. Si les données sont valides, elles sont converties en objet `Utilisateur`
4. Si les données sont invalides, FastAPI renvoie automatiquement une erreur 422 avec des détails

### Tester avec la documentation interactive

Lancez votre serveur et allez sur `http://127.0.0.1:8000/docs`. Vous verrez votre route avec un formulaire généré automatiquement !

Essayez d'envoyer :

```json
{
  "nom": "Alice",
  "age": 25,
  "email": "alice@example.com"
}
```

**Réponse :** ✅ Succès !

Maintenant, essayez avec des données invalides :

```json
{
  "nom": "Bob",
  "age": "vingt-cinq",
  "email": "bob@example.com"
}
```

**Réponse :** ❌ Erreur ! Pydantic détecte que `"vingt-cinq"` n'est pas un nombre entier et renvoie un message d'erreur clair.

## Types de paramètres dans FastAPI

FastAPI supporte trois types de paramètres principaux. Comprendre la différence est essentiel.

### 1. Paramètres de chemin (Path Parameters)

Ce sont les paramètres qui font partie de l'URL elle-même :

```python
@app.get("/utilisateurs/{utilisateur_id}")
def lire_utilisateur(utilisateur_id: int):
    return {"utilisateur_id": utilisateur_id}
```

**Exemple d'URL :** `http://127.0.0.1:8000/utilisateurs/42`

Le `42` est un paramètre de chemin qui sera automatiquement converti en entier.

### 2. Paramètres de requête (Query Parameters)

Ce sont les paramètres qui viennent après le `?` dans l'URL :

```python
@app.get("/recherche")
def rechercher(q: str, limite: int = 10):
    return {
        "requete": q,
        "limite": limite
    }
```

**Exemple d'URL :** `http://127.0.0.1:8000/recherche?q=python&limite=5`

Les paramètres de requête sont séparés par `&`. Remarquez que `limite` a une valeur par défaut (`10`), ce qui le rend optionnel.

### 3. Corps de requête (Request Body)

Ce sont les données envoyées dans le corps de la requête (généralement en POST, PUT, PATCH) :

```python
class Article(BaseModel):
    titre: str
    contenu: str

@app.post("/articles")
def creer_article(article: Article):
    return {"article": article}
```

**Données envoyées :**
```json
{
  "titre": "Mon article",
  "contenu": "Contenu de l'article..."
}
```

## Validation avancée avec Pydantic

Pydantic offre de nombreuses fonctionnalités pour valider vos données de manière précise.

### Champs optionnels

Certains champs peuvent être optionnels :

```python
class Utilisateur(BaseModel):
    nom: str
    age: int
    email: str
    telephone: str | None = None  # Optionnel
```

Avec `str | None`, le champ `telephone` peut être :
- Une chaîne de caractères
- `None` (la valeur par défaut)
- Absent de la requête

### Valeurs par défaut

Vous pouvez définir des valeurs par défaut pour vos champs :

```python
class Article(BaseModel):
    titre: str
    contenu: str
    publie: bool = False  # Faux par défaut
    vues: int = 0         # 0 par défaut
```

Si le client n'envoie pas ces champs, les valeurs par défaut seront utilisées.

### Contraintes de validation

Pydantic permet d'ajouter des contraintes sur vos champs avec `Field` :

```python
from pydantic import BaseModel, Field

class Utilisateur(BaseModel):
    nom: str = Field(..., min_length=2, max_length=50)
    age: int = Field(..., ge=0, le=150)  # ge = greater or equal, le = less or equal
    email: str = Field(..., pattern=r'^[\w\.-]+@[\w\.-]+\.\w+$')
```

Explications :
- `...` signifie "requis" (pas de valeur par défaut)
- `min_length=2` : Minimum 2 caractères
- `max_length=50` : Maximum 50 caractères
- `ge=0` : Greater or Equal (≥ 0)
- `le=150` : Less or Equal (≤ 150)
- `pattern=...` : Expression régulière pour valider le format

### Descriptions pour la documentation

Ajoutez des descriptions pour améliorer votre documentation automatique :

```python
class Utilisateur(BaseModel):
    nom: str = Field(..., description="Le nom complet de l'utilisateur")
    age: int = Field(..., description="L'âge en années", ge=0, le=150)
    email: str = Field(..., description="L'adresse email de l'utilisateur")
```

Ces descriptions apparaîtront dans la documentation Swagger !

### Validation d'email avec EmailStr

Pydantic offre des types spéciaux pour certaines validations courantes :

```python
from pydantic import BaseModel, EmailStr

class Utilisateur(BaseModel):
    nom: str
    age: int
    email: EmailStr  # Valide automatiquement le format email !
```

**Note :** Pour utiliser `EmailStr`, vous devez installer `email-validator` :

```bash
pip install email-validator
```

### Autres types spéciaux Pydantic

```python
from pydantic import BaseModel, HttpUrl, conint, constr

class Profil(BaseModel):
    nom: constr(min_length=2, max_length=50)  # String avec contraintes
    age: conint(ge=0, le=150)                  # Integer avec contraintes
    site_web: HttpUrl                          # Valide les URLs
    email: EmailStr                            # Valide les emails
```

## Combinaison de différents types de paramètres

Vous pouvez combiner paramètres de chemin, de requête et de corps dans une même route :

```python
class ModificationUtilisateur(BaseModel):
    nom: str | None = None
    age: int | None = None

@app.put("/utilisateurs/{utilisateur_id}")
def modifier_utilisateur(
    utilisateur_id: int,                    # Paramètre de chemin
    notifier: bool = False,                 # Paramètre de requête
    utilisateur: ModificationUtilisateur    # Corps de requête
):
    return {
        "utilisateur_id": utilisateur_id,
        "modifications": utilisateur,
        "notification_envoyee": notifier
    }
```

**Exemple d'utilisation :**
- URL : `http://127.0.0.1:8000/utilisateurs/42?notifier=true`
- Corps :
```json
{
  "nom": "Alice Dupont",
  "age": 26
}
```

Vos modèles Pydantic peuvent contenir d'autres modèles :


## Listes et modèles

Vous pouvez avoir des listes d'objets :

```python
class Tag(BaseModel):

    titre: str
    contenu: str
    tags: list[Tag]  # Liste de tags
```

**Exemple de données :**
```json
{

  "contenu": "Contenu...",
  "tags": [
    {"nom": "python"},

```

Ou simplement une liste de chaînes :


**Exemple de données :**
```json

```

## Validation personnalisée avec validators

Pour des validations complexes, vous pouvez créer des validateurs personnalisés :

```python
from pydantic import BaseModel, field_validator

class Utilisateur(BaseModel):
    nom: str
    age: int
    mot_de_passe: str

    @field_validator('nom')
    @classmethod
    def nom_doit_contenir_espace(cls, v: str) -> str:
        if ' ' not in v:
            raise ValueError('Le nom doit contenir un espace (prénom et nom)')
        return v

    @field_validator('mot_de_passe')
    @classmethod
    def mot_de_passe_fort(cls, v: str) -> str:
        if len(v) < 8:
            raise ValueError('Le mot de passe doit contenir au moins 8 caractères')
        if not any(char.isdigit() for char in v):
            raise ValueError('Le mot de passe doit contenir au moins un chiffre')
        return v
```

Le décorateur `@field_validator` vous permet de définir des règles personnalisées pour chaque champ.

## Modèles de réponse

Vous pouvez également définir le format des réponses de votre API :

```python
class UtilisateurEntree(BaseModel):
    nom: str
    age: int
    email: EmailStr
    mot_de_passe: str

class UtilisateurSortie(BaseModel):
    id: int
    nom: str
    age: int
    email: EmailStr
    # Pas de mot_de_passe dans la sortie !

@app.post("/utilisateurs", response_model=UtilisateurSortie)
def creer_utilisateur(utilisateur: UtilisateurEntree):
    # Simuler la création avec un ID
    return {
        "id": 1,
        "nom": utilisateur.nom,
        "age": utilisateur.age,
        "email": utilisateur.email,
        "mot_de_passe": utilisateur.mot_de_passe  # Sera exclu de la réponse !
    }
```

Avec `response_model=UtilisateurSortie`, FastAPI :
1. Valide que la réponse correspond au modèle
2. Filtre automatiquement les champs non présents dans le modèle (comme le mot de passe)
3. Génère la documentation correcte

## Gestion des erreurs de validation

Quand Pydantic détecte une erreur de validation, FastAPI renvoie automatiquement une réponse HTTP 422 (Unprocessable Entity) avec des détails sur l'erreur.

### Exemple d'erreur de validation

Si vous envoyez :
```json
{
  "nom": "A",
  "age": -5,
  "email": "pas-un-email"
}
```

Avec ce modèle :
```python
class Utilisateur(BaseModel):
    nom: str = Field(..., min_length=2)
    age: int = Field(..., ge=0)
    email: EmailStr
```

FastAPI renverra :
```json
{
  "detail": [
    {
      "loc": ["body", "nom"],
      "msg": "ensure this value has at least 2 characters",
      "type": "value_error.any_str.min_length"
    },
    {
      "loc": ["body", "age"],
      "msg": "ensure this value is greater than or equal to 0",
      "type": "value_error.number.not_ge"
    },
    {
      "loc": ["body", "email"],
      "msg": "value is not a valid email address",
      "type": "value_error.email"
    }
  ]
}
```

Chaque erreur indique :
- `loc` : L'emplacement du champ erroné
- `msg` : Le message d'erreur
- `type` : Le type d'erreur

## Paramètres de requête avec validation

Les paramètres de requête peuvent aussi bénéficier de la validation Pydantic :

```python
from fastapi import FastAPI, Query

app = FastAPI()

@app.get("/recherche")
def rechercher(
    q: str = Query(..., min_length=3, max_length=50),
    limite: int = Query(10, ge=1, le=100),
    page: int = Query(1, ge=1)
):
    return {
        "requete": q,
        "limite": limite,
        "page": page
    }
```

Avec `Query`, vous pouvez :
- Définir des valeurs par défaut
- Ajouter des contraintes de validation
- Rendre des paramètres obligatoires avec `...`
- Ajouter des descriptions

### Paramètres de requête optionnels

```python
@app.get("/recherche")
def rechercher(
    q: str | None = None,
    categorie: str | None = None,
    limite: int = 10
):
    resultats = {"limite": limite}
    if q:
        resultats["requete"] = q
    if categorie:
```

## Exemple complet : API de gestion d'articles

Voici un exemple complet qui combine tout ce que nous avons vu :

```python
from fastapi import FastAPI, HTTPException, Query, Path  
from pydantic import BaseModel, Field, EmailStr  
from datetime import datetime  

app = FastAPI(
    title="API de Blog",
    description="Une API pour gérer un blog",
    version="1.0.0"
)

# Modèles Pydantic
class Auteur(BaseModel):
    nom: str = Field(..., min_length=2, max_length=100)
    email: EmailStr

class ArticleBase(BaseModel):
    titre: str = Field(..., min_length=5, max_length=200)
    contenu: str = Field(..., min_length=10)
    tags: list[str] = []
    publie: bool = False

class ArticleCreation(ArticleBase):
    auteur: Auteur

class Article(ArticleBase):
    id: int
    auteur: Auteur
    date_creation: datetime
    vues: int = 0

# Simulation d'une base de données
articles_db = []  
prochain_id = 1  

# Routes
@app.get("/")
def lire_racine():
    return {"message": "Bienvenue sur l'API de Blog"}

@app.post("/articles", response_model=Article, status_code=201)
def creer_article(article: ArticleCreation):
    """Créer un nouvel article"""
    global prochain_id

    nouvel_article = {
        "id": prochain_id,
        "titre": article.titre,
        "contenu": article.contenu,
        "tags": article.tags,
        "publie": article.publie,
        "auteur": article.auteur.model_dump(),
        "date_creation": datetime.now(),
        "vues": 0
    }

    articles_db.append(nouvel_article)
    prochain_id += 1

    return nouvel_article

@app.get("/articles", response_model=list[Article])
def lire_articles(
    publies_seulement: bool = Query(False, description="Afficher uniquement les articles publiés"),
    limite: int = Query(10, ge=1, le=100, description="Nombre d'articles à retourner"),
    tag: str | None = Query(None, description="Filtrer par tag")
):
    """Récupérer la liste des articles"""
    resultats = articles_db.copy()

    # Filtrer par statut de publication
    if publies_seulement:
        resultats = [a for a in resultats if a["publie"]]

    # Filtrer par tag
    if tag:
        resultats = [a for a in resultats if tag in a["tags"]]

    # Limiter les résultats
    return resultats[:limite]

@app.get("/articles/{article_id}", response_model=Article)
def lire_article(
    article_id: int = Path(..., ge=1, description="L'ID de l'article")
):
    """Récupérer un article spécifique"""
    for article in articles_db:
        if article["id"] == article_id:
            # Incrémenter le nombre de vues
            article["vues"] += 1
            return article

    # Article non trouvé
    raise HTTPException(status_code=404, detail="Article non trouvé")

@app.put("/articles/{article_id}", response_model=Article)
def modifier_article(
    article_id: int,
    article_modifie: ArticleCreation
):
    """Modifier un article existant"""
    for index, article in enumerate(articles_db):
        if article["id"] == article_id:
            # Conserver certaines informations
            articles_db[index] = {
                "id": article_id,
                "titre": article_modifie.titre,
                "contenu": article_modifie.contenu,
                "tags": article_modifie.tags,
                "publie": article_modifie.publie,
                "auteur": article_modifie.auteur.model_dump(),
                "date_creation": article["date_creation"],  # Conserver la date originale
                "vues": article["vues"]  # Conserver les vues
            }
            return articles_db[index]

    raise HTTPException(status_code=404, detail="Article non trouvé")

@app.delete("/articles/{article_id}")
def supprimer_article(article_id: int):
    """Supprimer un article"""
    for index, article in enumerate(articles_db):
        if article["id"] == article_id:
            articles_db.pop(index)
            return {"message": f"Article {article_id} supprimé avec succès"}

    raise HTTPException(status_code=404, detail="Article non trouvé")

@app.get("/articles/{article_id}/statistiques")
def statistiques_article(article_id: int):
    """Obtenir les statistiques d'un article"""
    for article in articles_db:
        if article["id"] == article_id:
            return {
                "article_id": article_id,
                "titre": article["titre"],
                "vues": article["vues"],
                "nombre_tags": len(article["tags"]),
                "publie": article["publie"]
            }

    raise HTTPException(status_code=404, detail="Article non trouvé")
```

Lancez le serveur et allez sur `http://127.0.0.1:8000/docs`. Vous pouvez :

## Config et comportements avancés

### Exemples de données dans la documentation

Vous pouvez fournir des exemples qui apparaîtront dans la documentation :

```python
from pydantic import ConfigDict

class Utilisateur(BaseModel):
    model_config = ConfigDict(json_schema_extra={
        "examples": [
            {
                "nom": "Alice Dupont",
                "age": 25,
                "email": "alice@example.com"
            }
        ]
    })

    nom: str
    age: int
    email: EmailStr
```

### Alias pour les noms de champs

Parfois, les noms de champs doivent différer entre Python et JSON :

```python
from pydantic import Field

class Article(BaseModel):
    titre: str
    nombre_vues: int = Field(..., alias="nombreVues")
```

Maintenant, le JSON utilisera `nombreVues` mais votre code Python utilisera `nombre_vues`.

## Bonnes pratiques avec Pydantic

### 1. Séparez les modèles d'entrée et de sortie

```python
class UtilisateurCreation(BaseModel):
    nom: str
    email: EmailStr
    mot_de_passe: str

    nom: str
    email: EmailStr
```

```python
class Article(BaseModel):
    titre: str
    contenu: str
    publie: bool = False  # Par défaut non publié
```

```python
class Article(BaseModel):
```

### 4. Validez dès l'entrée

Ne laissez pas passer de données invalides. Utilisez les contraintes de validation.

### 5. Organisez vos modèles

Pour les gros projets, créez un fichier séparé `models.py` pour vos modèles Pydantic.
