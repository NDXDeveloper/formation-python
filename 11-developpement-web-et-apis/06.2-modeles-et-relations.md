# 11.6.2 Modèles et relations

## Introduction aux relations entre tables

Dans une base de données relationnelle, les tables sont souvent liées entre elles. Par exemple, un auteur peut écrire plusieurs livres, un étudiant peut s'inscrire à plusieurs cours, etc. SQLAlchemy facilite grandement la gestion de ces relations.

Dans ce chapitre, nous allons explorer les quatre types de relations principales :
- **One-to-Many** (un à plusieurs) : Un auteur peut avoir plusieurs livres
- **Many-to-One** (plusieurs à un) : Plusieurs livres appartiennent à un auteur
- **Many-to-Many** (plusieurs à plusieurs) : Un étudiant peut suivre plusieurs cours, et un cours peut avoir plusieurs étudiants
- **One-to-One** (un à un) : Un utilisateur a un seul profil

## Configuration initiale

Avant de commencer, voici la configuration de base que nous utiliserons dans tous nos exemples :

```python
from sqlalchemy import create_engine, Column, Integer, String, ForeignKey, Table  
from sqlalchemy.orm import declarative_base, relationship, sessionmaker  

# Configuration
Base = declarative_base()  
engine = create_engine('sqlite:///relations.db', echo=True)  
Session = sessionmaker(bind=engine)  
session = Session()  
```

## Relation One-to-Many (Un à Plusieurs)

C'est la relation la plus courante. Un enregistrement dans une table peut être lié à plusieurs enregistrements dans une autre table.

### Exemple : Auteur et Livres

Un auteur peut écrire plusieurs livres, mais chaque livre n'a qu'un seul auteur.

```python
class Auteur(Base):
    __tablename__ = 'auteurs'

    id = Column(Integer, primary_key=True)
    nom = Column(String(100), nullable=False)
    nationalite = Column(String(50))

    # Relation : un auteur a plusieurs livres
    # back_populates crée une liaison bidirectionnelle
    livres = relationship("Livre", back_populates="auteur")

    def __repr__(self):
        return f"<Auteur(nom='{self.nom}')>"


class Livre(Base):
    __tablename__ = 'livres'

    id = Column(Integer, primary_key=True)
    titre = Column(String(200), nullable=False)
    annee_publication = Column(Integer)

    # Clé étrangère : lie chaque livre à un auteur
    auteur_id = Column(Integer, ForeignKey('auteurs.id'))

    # Relation : plusieurs livres appartiennent à un auteur
    auteur = relationship("Auteur", back_populates="livres")

    def __repr__(self):
        return f"<Livre(titre='{self.titre}')>"
```

**Explication détaillée :**

- `ForeignKey('auteurs.id')` : Crée une clé étrangère qui référence la colonne `id` de la table `auteurs`
- `relationship("Livre", ...)` : Définit la relation au niveau Python (pas au niveau SQL)
- `back_populates` : Crée une liaison bidirectionnelle automatique entre les deux modèles

### Utilisation pratique

```python
# Création des tables
Base.metadata.create_all(engine)

# Création d'un auteur
victor_hugo = Auteur(nom="Victor Hugo", nationalite="Française")

# Création de livres
livre1 = Livre(titre="Les Misérables", annee_publication=1862)  
livre2 = Livre(titre="Notre-Dame de Paris", annee_publication=1831)  
livre3 = Livre(titre="Les Contemplations", annee_publication=1856)  

# Association des livres à l'auteur (méthode 1)
victor_hugo.livres.append(livre1)  
victor_hugo.livres.append(livre2)  
victor_hugo.livres.append(livre3)  

# Sauvegarde
session.add(victor_hugo)  
session.commit()  

print(f"Auteur créé avec {len(victor_hugo.livres)} livres")
```

**Méthode alternative :**

```python
# Création d'un auteur
moliere = Auteur(nom="Molière", nationalite="Française")  
session.add(moliere)  
session.commit()  

# Création de livres en assignant directement l'auteur (méthode 2)
livre4 = Livre(
    titre="Le Malade imaginaire",
    annee_publication=1673,
    auteur=moliere  # Association directe
)
livre5 = Livre(
    titre="Tartuffe",
    annee_publication=1664,
    auteur_id=moliere.id  # Ou en utilisant l'ID
)

session.add_all([livre4, livre5])  
session.commit()  
```

### Récupération des données liées

```python
# Récupérer un auteur et ses livres
auteur = session.query(Auteur).filter(Auteur.nom == "Victor Hugo").first()  
print(f"\n{auteur.nom} a écrit {len(auteur.livres)} livres :")  
for livre in auteur.livres:  
    print(f"  - {livre.titre} ({livre.annee_publication})")

# Récupérer un livre et son auteur
livre = session.query(Livre).filter(Livre.titre == "Les Misérables").first()  
print(f"\n'{livre.titre}' a été écrit par {livre.auteur.nom}")  
```

## Relation Many-to-One (Plusieurs à Un)

C'est l'inverse de One-to-Many. Techniquement, nous l'avons déjà vue dans l'exemple précédent (plusieurs livres → un auteur). La différence est principalement une question de perspective.

### Exemple : Commentaires et Articles

Plusieurs commentaires peuvent être associés à un seul article.

```python
class Article(Base):
    __tablename__ = 'articles'

    id = Column(Integer, primary_key=True)
    titre = Column(String(200), nullable=False)
    contenu = Column(String)

    # Un article a plusieurs commentaires
    commentaires = relationship("Commentaire", back_populates="article")

    def __repr__(self):
        return f"<Article(titre='{self.titre}')>"


class Commentaire(Base):
    __tablename__ = 'commentaires'

    id = Column(Integer, primary_key=True)
    texte = Column(String(500), nullable=False)
    auteur = Column(String(100))

    # Plusieurs commentaires appartiennent à un article
    article_id = Column(Integer, ForeignKey('articles.id'))
    article = relationship("Article", back_populates="commentaires")

    def __repr__(self):
        return f"<Commentaire(auteur='{self.auteur}')>"
```

### Utilisation

```python
# Création d'un article
article = Article(
    titre="Introduction à SQLAlchemy",
    contenu="SQLAlchemy est un ORM puissant..."
)

# Ajout de commentaires
commentaire1 = Commentaire(texte="Excellent article !", auteur="Alice")  
commentaire2 = Commentaire(texte="Très instructif", auteur="Bob")  
commentaire3 = Commentaire(texte="Merci pour le partage", auteur="Claire")  

# Association
article.commentaires.extend([commentaire1, commentaire2, commentaire3])

session.add(article)  
session.commit()  

# Affichage
print(f"\nArticle : {article.titre}")  
print(f"Nombre de commentaires : {len(article.commentaires)}")  
for comm in article.commentaires:  
    print(f"  - {comm.auteur} : {comm.texte}")
```

## Relation Many-to-Many (Plusieurs à Plusieurs)

Cette relation est plus complexe. Elle nécessite une **table d'association** (aussi appelée table de jonction) pour lier les deux tables principales.

### Exemple : Étudiants et Cours

Un étudiant peut s'inscrire à plusieurs cours, et un cours peut avoir plusieurs étudiants.

```python
# Table d'association (table intermédiaire)
# Cette table n'a pas besoin d'être une classe, juste une Table
inscription = Table(
    'inscriptions',
    Base.metadata,
    Column('etudiant_id', Integer, ForeignKey('etudiants.id'), primary_key=True),
    Column('cours_id', Integer, ForeignKey('cours.id'), primary_key=True)
)


class Etudiant(Base):
    __tablename__ = 'etudiants'

    id = Column(Integer, primary_key=True)
    nom = Column(String(100), nullable=False)
    prenom = Column(String(100), nullable=False)
    email = Column(String(100), unique=True)

    # Relation Many-to-Many avec les cours
    # secondary indique la table d'association à utiliser
    cours = relationship("Cours", secondary=inscription, back_populates="etudiants")

    def __repr__(self):
        return f"<Etudiant(nom='{self.nom} {self.prenom}')>"


class Cours(Base):
    __tablename__ = 'cours'

    id = Column(Integer, primary_key=True)
    code = Column(String(20), unique=True, nullable=False)
    intitule = Column(String(200), nullable=False)
    credits = Column(Integer)

    # Relation Many-to-Many avec les étudiants
    etudiants = relationship("Etudiant", secondary=inscription, back_populates="cours")

    def __repr__(self):
        return f"<Cours(code='{self.code}', intitule='{self.intitule}')>"
```

**Explication de la table d'association :**

- `inscription` : Table intermédiaire qui stocke les liens entre étudiants et cours
- Elle contient deux clés étrangères (une vers `etudiants`, une vers `cours`)
- Les deux colonnes forment ensemble la clé primaire (pas de duplicata)
- `secondary=inscription` : Indique à SQLAlchemy quelle table utiliser pour la liaison

### Utilisation

```python
# Création des tables
Base.metadata.create_all(engine)

# Création d'étudiants
etudiant1 = Etudiant(nom="Durand", prenom="Sophie", email="sophie.durand@school.com")  
etudiant2 = Etudiant(nom="Martin", prenom="Lucas", email="lucas.martin@school.com")  
etudiant3 = Etudiant(nom="Bernard", prenom="Emma", email="emma.bernard@school.com")  

# Création de cours
cours1 = Cours(code="MATH101", intitule="Mathématiques avancées", credits=6)  
cours2 = Cours(code="INFO202", intitule="Programmation Python", credits=5)  
cours3 = Cours(code="PHYS150", intitule="Physique quantique", credits=4)  

# Inscription des étudiants aux cours
# Sophie s'inscrit à Math et Python
etudiant1.cours.append(cours1)  
etudiant1.cours.append(cours2)  

# Lucas s'inscrit à Python et Physique
etudiant2.cours.extend([cours2, cours3])

# Emma s'inscrit aux trois cours
etudiant3.cours = [cours1, cours2, cours3]

# Sauvegarde
session.add_all([etudiant1, etudiant2, etudiant3])  
session.commit()  

# Affichage des inscriptions
print("\n=== Inscriptions par étudiant ===")  
for etudiant in session.query(Etudiant).all():  
    print(f"\n{etudiant.prenom} {etudiant.nom} :")
    for cours in etudiant.cours:
        print(f"  - {cours.code} : {cours.intitule}")

print("\n=== Étudiants par cours ===")  
for cours in session.query(Cours).all():  
    print(f"\n{cours.code} - {cours.intitule} ({len(cours.etudiants)} étudiants) :")
    for etudiant in cours.etudiants:
        print(f"  - {etudiant.prenom} {etudiant.nom}")
```

### Manipulation des relations Many-to-Many

```python
# Ajouter un cours à un étudiant
sophie = session.query(Etudiant).filter(Etudiant.prenom == "Sophie").first()  
physique = session.query(Cours).filter(Cours.code == "PHYS150").first()  
sophie.cours.append(physique)  
session.commit()  

# Retirer un cours d'un étudiant
sophie.cours.remove(physique)  
session.commit()  

# Vérifier si un étudiant est inscrit à un cours
if cours2 in etudiant2.cours:
    print("Lucas est inscrit au cours de Python")
```

## Relation One-to-One (Un à Un)

Dans cette relation, un enregistrement d'une table correspond à un seul enregistrement d'une autre table.

### Exemple : Utilisateur et Profil

Chaque utilisateur a un seul profil, et chaque profil appartient à un seul utilisateur.

```python
class Utilisateur(Base):
    __tablename__ = 'utilisateurs'

    id = Column(Integer, primary_key=True)
    username = Column(String(50), unique=True, nullable=False)
    email = Column(String(100), unique=True, nullable=False)

    # Relation One-to-One avec Profil
    # uselist=False indique qu'il s'agit d'une relation 1:1 (pas une liste)
    profil = relationship("Profil", back_populates="utilisateur", uselist=False)

    def __repr__(self):
        return f"<Utilisateur(username='{self.username}')>"


class Profil(Base):
    __tablename__ = 'profils'

    id = Column(Integer, primary_key=True)
    bio = Column(String(500))
    site_web = Column(String(200))
    date_naissance = Column(String(20))

    # Clé étrangère vers utilisateur (unique=True pour garantir 1:1)
    utilisateur_id = Column(Integer, ForeignKey('utilisateurs.id'), unique=True)

    # Relation inverse
    utilisateur = relationship("Utilisateur", back_populates="profil")

    def __repr__(self):
        return f"<Profil(utilisateur='{self.utilisateur.username if self.utilisateur else None}')>"
```

**Point clé :** `uselist=False` transforme la relation en One-to-One. Sans cela, `utilisateur.profil` serait une liste.

### Utilisation

```python
# Création d'un utilisateur avec son profil
user = Utilisateur(username="alice_dev", email="alice@dev.com")  
profil = Profil(  
    bio="Développeuse Python passionnée",
    site_web="https://alice-dev.com",
    date_naissance="15/03/1990"
)

# Association
user.profil = profil

session.add(user)  
session.commit()  

# Accès aux données
print(f"\nUtilisateur : {user.username}")  
print(f"Bio : {user.profil.bio}")  
print(f"Site web : {user.profil.site_web}")  

# Sens inverse
profil_recupere = session.query(Profil).first()  
print(f"\nCe profil appartient à : {profil_recupere.utilisateur.username}")  
```

## Relations en cascade

Les cascades permettent de définir ce qui se passe avec les objets liés lors de certaines opérations (suppression, mise à jour, etc.).

### Options de cascade courantes

```python
class Auteur(Base):
    __tablename__ = 'auteurs_cascade'

    id = Column(Integer, primary_key=True)
    nom = Column(String(100))

    # Cascade : si on supprime l'auteur, ses livres sont aussi supprimés
    livres = relationship(
        "Livre",
        back_populates="auteur",
        cascade="all, delete-orphan"
    )


class Livre(Base):
    __tablename__ = 'livres_cascade'

    id = Column(Integer, primary_key=True)
    titre = Column(String(200))
    auteur_id = Column(Integer, ForeignKey('auteurs_cascade.id'))
    auteur = relationship("Auteur", back_populates="livres")
```

**Options de cascade :**

- `all` : Propage toutes les opérations (save, merge, refresh, expunge, delete)
- `delete` : Supprime les objets liés quand l'objet parent est supprimé
- `delete-orphan` : Supprime les objets liés s'ils ne sont plus associés au parent
- `save-update` : Propage les ajouts et modifications
- `merge` : Propage les fusions d'objets
- `refresh` : Propage les rafraîchissements
- `expunge` : Propage les expulsions de session

### Exemple pratique

```python
# Sans cascade delete-orphan
auteur = session.query(Auteur).first()  
livre = auteur.livres[0]  

# Retirer le livre de la liste (mais il reste en base)
auteur.livres.remove(livre)  
session.commit()  
# Le livre existe toujours en base, mais sans auteur

# Avec cascade delete-orphan
auteur = session.query(Auteur).first()  
livre = auteur.livres[0]  

# Retirer le livre de la liste
auteur.livres.remove(livre)  
session.commit()  
# Le livre est automatiquement supprimé de la base
```

## Chargement des relations (Lazy Loading vs Eager Loading)

Par défaut, SQLAlchemy utilise le "lazy loading" : les objets liés ne sont chargés que lorsqu'on y accède.

### Lazy Loading (par défaut)

```python
# Récupération d'un auteur
auteur = session.query(Auteur).first()
# À ce stade, les livres ne sont pas encore chargés

# Accès aux livres → déclenche une nouvelle requête SQL
for livre in auteur.livres:  # Requête SQL ici !
    print(livre.titre)
```

**Problème potentiel :** Le "N+1 query problem" - si vous parcourez 100 auteurs et leurs livres, vous ferez 101 requêtes SQL (1 pour les auteurs + 100 pour les livres).

### Eager Loading (chargement anticipé)

Pour éviter le problème N+1, utilisez `joinedload` ou `selectinload` :

```python
from sqlalchemy.orm import joinedload, selectinload

# Méthode 1 : joinedload (utilise un JOIN SQL)
auteurs = session.query(Auteur).options(joinedload(Auteur.livres)).all()
# Une seule requête SQL avec JOIN

# Méthode 2 : selectinload (utilise un SELECT IN)
auteurs = session.query(Auteur).options(selectinload(Auteur.livres)).all()
# Deux requêtes SQL optimisées

# Maintenant, parcourir les livres ne génère pas de requêtes supplémentaires
for auteur in auteurs:
    print(f"\n{auteur.nom} :")
    for livre in auteur.livres:  # Pas de requête SQL !
        print(f"  - {livre.titre}")
```

**Configuration au niveau du modèle :**

```python
class Auteur(Base):
    __tablename__ = 'auteurs'
    id = Column(Integer, primary_key=True)
    nom = Column(String(100))

    # Configuration du chargement par défaut
    livres = relationship(
        "Livre",
        back_populates="auteur",
        lazy="select"  # Options: 'select', 'joined', 'selectin', 'subquery'
    )
```

## Relations auto-référentielles

Une table peut avoir une relation avec elle-même. Exemple typique : employés et managers.

```python
class Employe(Base):
    __tablename__ = 'employes'

    id = Column(Integer, primary_key=True)
    nom = Column(String(100), nullable=False)
    poste = Column(String(100))

    # Clé étrangère vers le manager (qui est aussi un employé)
    manager_id = Column(Integer, ForeignKey('employes.id'))

    # Relation vers le manager
    manager = relationship("Employe", remote_side=[id], backref="equipe")

    def __repr__(self):
        return f"<Employe(nom='{self.nom}', poste='{self.poste}')>"
```

**Explication :**
- `remote_side=[id]` : Indique le côté "distant" de la relation
- `backref="equipe"` : Crée automatiquement l'attribut `equipe` pour accéder aux subordonnés

### Utilisation

```python
# Création de la hiérarchie
ceo = Employe(nom="Directrice Générale", poste="CEO")

manager1 = Employe(nom="Chef Ventes", poste="Manager", manager=ceo)  
manager2 = Employe(nom="Chef IT", poste="Manager", manager=ceo)  

employe1 = Employe(nom="Vendeur 1", poste="Commercial", manager=manager1)  
employe2 = Employe(nom="Vendeur 2", poste="Commercial", manager=manager1)  
employe3 = Employe(nom="Développeur", poste="Developer", manager=manager2)  

session.add(ceo)  
session.commit()  

# Affichage de la hiérarchie
print(f"\n{ceo.nom} manage :")  
for manager in ceo.equipe:  
    print(f"  - {manager.nom} ({manager.poste})")
    if manager.equipe:
        for employe in manager.equipe:
            print(f"    - {employe.nom} ({employe.poste})")
```

## Association Object (Table d'association avec attributs)

Parfois, la table d'association elle-même a besoin d'attributs supplémentaires. Dans ce cas, elle doit être une vraie classe.

### Exemple : Étudiants et Cours avec notes

```python
class Inscription(Base):
    __tablename__ = 'inscriptions_notes'

    etudiant_id = Column(Integer, ForeignKey('etudiants.id'), primary_key=True)
    cours_id = Column(Integer, ForeignKey('cours.id'), primary_key=True)

    # Attributs supplémentaires
    note = Column(Integer)
    date_inscription = Column(String)
    statut = Column(String(20))  # 'en_cours', 'terminé', 'abandonné'

    # Relations
    etudiant = relationship("Etudiant", back_populates="inscriptions")
    cours = relationship("Cours", back_populates="inscriptions")

    def __repr__(self):
        return f"<Inscription(note={self.note}, statut='{self.statut}')>"


class Etudiant(Base):
    __tablename__ = 'etudiants_v2'

    id = Column(Integer, primary_key=True)
    nom = Column(String(100))
    prenom = Column(String(100))

    inscriptions = relationship("Inscription", back_populates="etudiant")


class Cours(Base):
    __tablename__ = 'cours_v2'

    id = Column(Integer, primary_key=True)
    code = Column(String(20))
    intitule = Column(String(200))

    inscriptions = relationship("Inscription", back_populates="cours")
```

### Utilisation

```python
# Création
etudiant = Etudiant(nom="Dupont", prenom="Marie")  
cours = Cours(code="MATH101", intitule="Mathématiques")  

# Inscription avec attributs supplémentaires
inscription = Inscription(
    etudiant=etudiant,
    cours=cours,
    note=15,
    date_inscription="2024-09-01",
    statut="en_cours"
)

session.add_all([etudiant, cours, inscription])  
session.commit()  

# Consultation
for insc in etudiant.inscriptions:
    print(f"{etudiant.prenom} a eu {insc.note}/20 en {insc.cours.intitule}")
```

## Bonnes pratiques

### 1. Nommage cohérent

```python
# Bon : noms clairs et cohérents
class Auteur(Base):
    livres = relationship("Livre", back_populates="auteur")

class Livre(Base):
    auteur = relationship("Auteur", back_populates="livres")
```

### 2. Utiliser back_populates plutôt que backref

```python
# Préférable : explicite et bidirectionnel
livres = relationship("Livre", back_populates="auteur")

# Moins préférable : implicite (mais fonctionne)
livres = relationship("Livre", backref="auteur")
```

### 3. Éviter les boucles infinies dans __repr__

```python
# Mauvais : peut causer une boucle infinie
def __repr__(self):
    return f"<Auteur(nom='{self.nom}', livres={self.livres})>"

# Bon : n'inclut pas les objets liés
def __repr__(self):
    return f"<Auteur(nom='{self.nom}')>"
```

### 4. Utiliser des contraintes de base de données

```python
# Bon : contrainte au niveau SQL
auteur_id = Column(Integer, ForeignKey('auteurs.id', ondelete='CASCADE'))
# ondelete='CASCADE' : supprime les livres si l'auteur est supprimé
# ondelete='SET NULL' : met auteur_id à NULL
# ondelete='RESTRICT' : empêche la suppression de l'auteur
```

## Exemple complet : Bibliothèque

Voici un exemple complet intégrant plusieurs types de relations :

```python
from sqlalchemy import create_engine, Column, Integer, String, ForeignKey, Table  
from sqlalchemy.orm import declarative_base, relationship, sessionmaker  

Base = declarative_base()  
engine = create_engine('sqlite:///bibliotheque.db', echo=False)  
Session = sessionmaker(bind=engine)  

# Table d'association pour Many-to-Many
emprunt_livre = Table(
    'emprunts_livres',
    Base.metadata,
    Column('emprunt_id', Integer, ForeignKey('emprunts.id')),
    Column('livre_id', Integer, ForeignKey('livres.id'))
)


class Bibliotheque(Base):
    __tablename__ = 'bibliotheques'

    id = Column(Integer, primary_key=True)
    nom = Column(String(100), nullable=False)
    adresse = Column(String(200))

    # One-to-Many : une bibliothèque a plusieurs livres
    livres = relationship("Livre", back_populates="bibliotheque")


class Livre(Base):
    __tablename__ = 'livres'

    id = Column(Integer, primary_key=True)
    titre = Column(String(200), nullable=False)
    isbn = Column(String(20), unique=True)

    # Many-to-One : plusieurs livres appartiennent à une bibliothèque
    bibliotheque_id = Column(Integer, ForeignKey('bibliotheques.id'))
    bibliotheque = relationship("Bibliotheque", back_populates="livres")

    # Many-to-One : plusieurs livres d'un auteur
    auteur_id = Column(Integer, ForeignKey('auteurs.id'))
    auteur = relationship("Auteur", back_populates="livres")

    # Many-to-Many : un livre peut être dans plusieurs emprunts
    emprunts = relationship("Emprunt", secondary=emprunt_livre, back_populates="livres")


class Auteur(Base):
    __tablename__ = 'auteurs'

    id = Column(Integer, primary_key=True)
    nom = Column(String(100), nullable=False)
    nationalite = Column(String(50))

    # One-to-Many : un auteur a plusieurs livres
    livres = relationship("Livre", back_populates="auteur")


class Membre(Base):
    __tablename__ = 'membres'

    id = Column(Integer, primary_key=True)
    nom = Column(String(100), nullable=False)
    email = Column(String(100), unique=True)

    # One-to-Many : un membre a plusieurs emprunts
    emprunts = relationship("Emprunt", back_populates="membre")


class Emprunt(Base):
    __tablename__ = 'emprunts'

    id = Column(Integer, primary_key=True)
    date_emprunt = Column(String(20), nullable=False)
    date_retour_prevue = Column(String(20))

    # Many-to-One : plusieurs emprunts pour un membre
    membre_id = Column(Integer, ForeignKey('membres.id'))
    membre = relationship("Membre", back_populates="emprunts")

    # Many-to-Many : un emprunt peut contenir plusieurs livres
    livres = relationship("Livre", secondary=emprunt_livre, back_populates="emprunts")


# Création de la base de données
Base.metadata.create_all(engine)

# Utilisation
session = Session()

# Création d'une bibliothèque
bibli = Bibliotheque(nom="Bibliothèque Municipale", adresse="12 rue de la Paix")

# Création d'auteurs
hugo = Auteur(nom="Victor Hugo", nationalite="Française")

# Création de livres
livre1 = Livre(titre="Les Misérables", isbn="978-0-1234-5678-9", auteur=hugo, bibliotheque=bibli)

# Création d'un emprunt

    membre=membre

emprunt.livres.extend([livre1, livre3])

# Sauvegarde
session.add_all([bibli, hugo, dumas, membre, emprunt])
```

## Résumé

**Types de relations :**
- **One-to-Many** : `ForeignKey` dans la table "Many" + `relationship` des deux côtés
- **Many-to-One** : Perspective inverse de One-to-Many
- **Many-to-Many** : Table d'association + `secondary` dans `relationship`
- **One-to-One** : Comme One-to-Many mais avec `uselist=False`

**Concepts clés :**
- `ForeignKey` : Définit la contrainte au niveau SQL
- `relationship` : Définit la relation au niveau Python ORM
- `back_populates` : Crée une liaison bidirectionnelle explicite
- `cascade` : Contrôle la propagation des opérations
- Lazy vs Eager loading : Contrôle du moment de chargement des données liées

**Bonnes pratiques :**
- Utiliser `back_populates` pour plus de clarté
- Éviter les boucles infinies dans `__repr__`
- Optimiser avec eager loading si nécessaire
- Utiliser les contraintes SQL (`ondelete`, etc.)

Vous maîtrisez maintenant les relations dans SQLAlchemy ! Ces concepts vous permettent de modéliser des structures de données complexes et de naviguer facilement entre les objets liés.
